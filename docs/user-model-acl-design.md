# 用户模型权限控制系统设计 (User-Model ACL System Design)

## 1. 业务目标与背景
ParaRouter 需要提供一种轻量、灵活且安全的机制，允许系统管理员和终端用户对大模型 API 的调用进行权限与消费额度控制。
现有系统仅在“账户大盘水位”上进行宏观充值与扣费，所有的在线模型对所有账户都开放。本设计旨在解决以下痛点：
1. **分销管控**：管理员希望在（代）注册用户时，能够定死给特定客户售卖的具体模型范围（例如，仅限普通模型，禁止使用高价的 GPT-4）。
2. **客户自助风控**：具有一定开发领域的终端用户，希望能在自己的项目中生成子级 API Key 并设定限额流控，以防密钥泄露导致的异常账单。
3. **差异化定价**：管理员希望避免繁重的“用户分组（User Group）”管理，能一键针对某位熟人客户修改全局请求单价折扣。

## 2. 核心架构：“底层强控” + “表层自管”

为了同时满足“Admin 强管控”、“代注册极简体验”以及“C端自助化”的三方面需求，我们在架构上避免介入复杂的 Role/Group 设计，采用两级扁平化的字段覆写（Field Override）方案：

### 2.1 用户级 (User Level) - Admin 强控边界
在代表最高权属的 `users` 账户级别确立“绝对边界”。由系统管理员负责维护（无论通过 Admin 后台调整，还是代注册时设定）。
- **`allowed_models` (允许模型范围)**：存储该账号有权访问的模型 ID 数组。如果该字段为 NULL或空，则表示享有平台全局模型访问权。即使该用户在前端自行创建 API Key，也绝无可能逾越由管理员通过 `allowed_models` 划定的模型许可。
- **`pricing_multiplier` (计费倍率)**：提供针对特定账户的“定制优惠”或“杀熟”费率（默认为 `1.0`）。网关在结算每次真实请求时，统一以 `平台标准模型价 * 用户倍率` 扣除 `balance`。

### 2.2 密钥级 (API Key Level) - 用户自管流水
保留类 OpenRouter 的极致开发者体验，终端用户只需对分配给自己的 API Key 负责。
- **`budget_limit` (密钥消耗上限)**：限制此特定令牌的生命周期内最大允许消费总额（美元）。防止单一 Key 泄露造成的资金全损。
- *(可选)* **`allowed_models` (子集限制)**：在不超过主账户所属模型边界的前提下，用户还可以进一步将特定 API Key 局限在仅限调用某几个普通模型。

## 3. 业务链路工作流

### 3.1 GTM / 销售“代管理”流程
1. **一键发号**：销售或 Admin 与企业客户沟通后，直接在系统 Admin 面板点击【新建客户】。
2. **界定权责**：在自动生成“账号、密码、首发 API Key”的同步窗口中，下拉多选指定划拨给该客户的具体 API 模型组合（如仅勾选 Claude-3 模型），并填入双方约定的 `pricing_multiplier` 折扣费率。
3. **极简交付**：系统底层同时生成 `users` 行和 `user_api_keys` 列。Admin 随手将账密和 `sk-xxx` 丢给客户，即可立刻使用。
4. **系统级防穿透**：由于模型权限死锁在其底层 `users` ID 上，该客户拿到密码就算想耍小聪明，登录平台去自己创建新的无限权限 API Key，其全部请求仍将严格被限制在 Claude-3 系列内。体系保持高安全水位。

### 3.2 极客散户自助注册流程
1. **0额度准入**：极客通过前台门户自主脱离销售完成注册入库，默认获得最高级别的跨平台调用权限 (`allowed_models` = NULL 开放态)，默认标准无折扣 (`pricing_multiplier` = 1.0)。但其账户初始测试金 `balance` = 0。
2. **引流对接**：极客迫不及待发起任何首笔 LLM 请求，都会立刻撞上网关风控边界：HTTP 402 "Insufficient balance / 余额不足"。
3. **资金充值**：用户扫码加群/私敲客服完成支付补款，Admin 手动在其账号加上资金即算激活。其后的测试过程中，极客可利用前台“我的 API Keys”功能自主去切分 `budget_limit` 控制各个子项开销，系统体验极佳。
